# Worker Batch Task-Step Functionality — Implementation Intention
## Goal
Implement frontend support for batch-capable working sections in the worker app.
The backend now supports:
1. Marking a working section as batch-capable using `allows_batch_working`.
2. Returning an active batch group for the worker resume card through `active_batch_steps`.
3. Transitioning multiple batch-capable task steps atomically through the new batch transition endpoint.
The frontend should use these backend changes to allow workers to select, start, pause/resume, and complete multiple task steps as a batch while still preserving individual task-step interaction where needed.
This implementation should be clean, scalable, and aligned with the existing worker app architecture.

Realtime behavior

The batch transition handoff says:

* The backend broadcasts one task:step-state-changed BatchWorkspaceEvent.
* The event contains an items array of { client_id, new_state }.
* Do not expect one event per changed step.
* There may also be one task:state-changed event per distinct affected task.

Frontend realtime handling must support this coalesced event shape.

Claude should inspect the existing task-step realtime handlers and update them so that:

* single-step events still work;
* batch events update/refetch all affected steps;
* the UI does not assume exactly one changed step per event.

⸻

Error handling

Batch transition failures must be handled carefully because the backend is atomic.

If the backend returns:

{ "ok": false, "error": "Batch transition rejected — <step_id>: <reason>; ..." }

Frontend should:

* surface the error message to the worker;
* refetch the affected working-section steps;
* clear or preserve selection intentionally depending on UX;
* avoid optimistic UI that assumes partial success.

The backend guarantees:

* if one step fails validation, no step is mutated.

Therefore the UI should not show partial success after a failed batch call.

## Update: batch completion now supports inaccurate timing
The backend handoff `HANDOFF_TO_FRONTEND_batch_step_transition_20260623.md` has been updated.
The batch transition endpoint now supports inaccurate-time marking through a per-item field:
```ts
mark_closing_record_inaccurate?: boolean

This means batch completion can preserve the same core timing-confirmation UX as the existing single-step completion flow.

For batch completion:

* The confirmation flow may still ask whether the recorded time is accurate.
* If the worker marks the time as accurate, send each completed item without mark_closing_record_inaccurate, or with it set to false.
* If the worker marks the batch timing as inaccurate, send mark_closing_record_inaccurate: true on each batch item being completed.
* Only include steps currently in working state.
* The batch endpoint still uses one shared new_state, usually completed.
* The endpoint still does not support per-item credited_user_id or per-item new_state.

Example batch completion request with inaccurate timing:

{
  "items": [
    {
      "task_id": "tsk_01",
      "step_id": "tsp_07",
      "mark_closing_record_inaccurate": true
    },
    {
      "task_id": "tsk_02",
      "step_id": "tsp_03",
      "mark_closing_record_inaccurate": true
    }
  ],
  "new_state": "completed",
  "reason": null,
  "description": null
}
