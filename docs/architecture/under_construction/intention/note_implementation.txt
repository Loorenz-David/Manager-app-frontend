INTENTION:

We will now create the implemenation for task notes.

First we must update the task schema on the manager application for task, because the task_notes no longer come in the payload.

all the following task note components and logic will be build as task feature packages for the manager, worker and other applications to consume. contract 35 describes how to build packages for applications to consume, including how page sheets should be consume by application surfaces.
if the implementation deserves it's own feature folder in packages we can do that, to not cluster the task feature package.

the first component we will create is the task note composer, this composer will follow the same architecture as the CaseInitialMessageComposer because the user can write notes with styled content.
but this composer will have a button at the bottom right corner of a camera icon, allowing the user to trigger the camera to take pictures for the note like the CaseRichComposer which renders the CaseComposerInlineCameraButton
pictures taken from the note composer are display in a image grid above the case composer container, a visual max of 3 pictures previews, after that we render a dark overlay on the third picture conteinar with a label "+{number of extra picture}". like the CaseComposerDraftImagesProvider at the CaseRichComposer. Allowing the user to remove the image or open it in full image page.
the note composer and the image preview grid are separate the major component using the components decides how to arrange this two components

the note composer should allow for the parent to inject an already create note content because we will use the note composer also for editing already created notes.

At the task creation forms InternalFormContent, PreOrderFormContent, ReturnFormContent, we will replace the current field TaskAdditionalDetailsField for this new note composer.
for the images taken from the note composer they will render on the ImagePreviewGrid of the forms. no need for the note image grid preview here. 

beacuse the frontend controlls the client_ids use for creating images and tasks the forms should create a client_id for the note being created at the form using the client id utitily, same client id it will use to create the image, like the current task image system does.
on submit it will send the note object with the client id created if there is an image for note created or if the note content has characters. 

the endpoint we use for creating the notes through the form is the same endpoint we use for creating the task, the task endpoint has made modifications to accept this new task note shape in it's payload, this is documented at the handoff HANDOFF_TO_FRONTEND_task_note_system_improvement_20260626
