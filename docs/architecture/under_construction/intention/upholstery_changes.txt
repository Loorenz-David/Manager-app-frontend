The primary issue is the sound file itself, not your visual changes.
What I verified:
The celebration preset is trying to play /sounds/celebration.mp3.
The overlay does call play(config.soundUrl) when the animation opens.
The actual file at [celebration.mp3](/Users/davidloorenz/Desktop/Developer/BeyoApps_2025/ManagerBeyo-app/frontend/apps/workers-app/ManagerBeyo-app-workers/public/sounds/celebration.mp3) is 0 bytes and file reports it as empty.
That means the browser is being asked to play an empty MP3, so there is no audible sound to play.
Why you don’t see an error:
[use-celebration-sound.ts](/Users/davidloorenz/Desktop/Developer/BeyoApps_2025/ManagerBeyo-app/frontend/packages/celebration/src/hooks/use-celebration-sound.ts) intentionally swallows audio.play() failures.
So missing/invalid audio fails silently by design.
There is also a secondary risk, but it is not the main issue here:
The sound is triggered after the completion mutation succeeds, so on some mobile browsers autoplay could be blocked because it is no longer in the original tap gesture.
But in your current setup, the concrete, confirmed problem is simpler: the target sound file is empty.
