Intention: Pending Seat Upholstery Management

1. Purpose

Implement a manager-facing workflow for finding and resolving seat tasks whose primary item is blocked by incomplete upholstery data.

The workflow must allow an authenticated user of the manager application to:

1. See seat tasks whose primary item has no upholstery selected.
2. See seat tasks whose selected upholstery has no required quantity in meters.
3. Search and filter those tasks.
4. Open the existing task-detail page.
5. Resolve the missing upholstery information directly from the pending list.
6. Keep the pending list, counts, home-screen badge, item-upholstery data, and task-detail state synchronized after mutations.

This feature is a frontend integration of the backend contract documented in:

frontend/docs/handoff/from_backend/HANDOFF_TO_FRONTEND_seat_tasks_pending_upholstery_20260615.md

The implementation must reuse the existing task, upholstery, surface, query, pull-to-refresh, image-viewer, and scroll-visibility architecture. It must not introduce duplicate upholstery-selection or quantity-editing flows.

⸻

2. Primary user flow

The user enters the pending upholstery page from the manager home screen.

The page initially displays tasks in the missing_selection bucket.

From the page, the user can:

* search using the backend q parameter;
* switch between missing_selection and missing_quantity;
* activate both filters to display the union of both buckets;
* pull down to refresh the current data;
* load additional pages;
* tap the task-card body to open task details;
* tap the task image to open the existing full-image preview;
* tap the three-dot action to open a temporary bottom-sheet menu containing Coming soon;
* tap a direct action below the task card to resolve the missing upholstery information;
* close the page using either the header back button or the footer Close & Back button.

The direct action depends on the row’s backend-provided classification:

* pending_upholstery_reason === "missing_selection" opens the existing upholstery picker slide page;
* pending_upholstery_reason === "missing_quantity" opens the existing upholstery amount bottom sheet.

The frontend must use pending_upholstery_reason as the source of truth. It must not infer the pending condition from the selected filter, the existence of an item, or other row properties.

After a successful mutation, a task that no longer matches the active pending condition should disappear immediately where that transition can be determined safely. All affected queries must then reconcile with the backend.

4. Backend contract

Pending task list

Endpoint:

GET /api/v1/item-upholsteries/pending-seat-tasks

Supported query parameters:

* limit: optional, default 50, maximum 200;
* offset: optional, default 0;
* q: optional, maximum length 200;
* missing_selection: optional boolean, default false;
* missing_quantity: optional boolean, default false.

Filter semantics:

* missing_selection=true returns tasks whose primary seat item has no active ItemUpholstery record.
* missing_quantity=true returns tasks whose primary seat item has an active upholstery record whose amount_meters is null or 0.
* Both filters false, or both filters true, return the union of both buckets.

The initial page request must explicitly send:

missing_selection=true
missing_quantity=false

Do not depend on the backend’s default combined result for the initial view.

List response data is located at:

data.tasks_pagination.items
data.tasks_pagination.limit
data.tasks_pagination.offset
data.tasks_pagination.has_more

Each tasks_pagination.items[] row contains:

task
primary_item
pending_upholstery_reason
item_images

pending_upholstery_reason has one of the following values:

missing_selection
missing_quantity

Meaning:

* missing_selection means the primary seat item has no active ItemUpholstery record.
* missing_quantity means the primary seat item has an active ItemUpholstery record, but its amount_meters is null or 0.

The frontend must use this field directly for:

* selecting the card’s direct action;
* action labels;
* reason-specific rendering;
* combined-filter results;
* optimistic list transitions;
* any future reason badge or grouping.

The frontend must not infer the reason from the currently active filter. This is especially important when both filters are active and the endpoint returns the union of both buckets.

primary_item must be handled as nullable.

The first image may use the complete serialized image shape while later images may use a lighter shape. Shared UI code may safely depend only on:

client_id
image_url
width_px
height_px
file_size_bytes

Pending counts

Endpoint:

GET /api/v1/item-upholsteries/pending-seat-tasks/counts

Response values:

data.missing_selection_total
data.missing_quantity_total

The counts are workspace-wide and independent of:

* search;
* pagination;
* the current list filters.

Use the counts endpoint for:

* filter-pill labels;
* the manager home-screen entry count;
* inexpensive reconciliation after upholstery mutations.

The home-screen total is:

missing_selection_total + missing_quantity_total

The backend contract defines these as mutually exclusive buckets, so summing them is valid.

Authentication and authorization

The endpoints require authentication.

Allowed roles:

ADMIN
MANAGER
WORKER

Other roles receive 403.

The frontend should follow the manager application’s existing authorization behavior rather than creating a feature-specific authorization mechanism.

5. Remaining backend contract question for the quantity action

The backend now provides pending_upholstery_reason, so the frontend can correctly classify every row, including rows returned from the combined union view.

The classification problem is therefore resolved.

However, a missing_quantity direct action still needs to identify the existing ItemUpholstery record that must be updated.

The existing amount-sheet flow appears to require:

itemUpholsteryId

The documented pending-list row currently exposes:

task
primary_item
pending_upholstery_reason
item_images

It does not document the existing active ItemUpholstery.client_id.

Claude must inspect:

* the actual backend serializer;
* generated or handwritten frontend API types;
* existing item-upholstery queries;
* the current upholstery amount-sheet contract;
* the current task-detail data already available to the amount flow.

The implementation plan must determine how the pending page obtains the existing item-upholstery identifier without introducing an N+1 request pattern.

Preferred backend contract, if the identifier is not already available elsewhere in the row:

type PendingSeatTaskItem = {
  task: Task;
  primary_item: Item | null;
  pending_upholstery_reason:
    | "missing_selection"
    | "missing_quantity";
  item_upholstery_id: string | null;
  item_images: PendingTaskImage[];
};

Expected rules:

missing_selection:
  item_upholstery_id = null
missing_quantity:
  item_upholstery_id = non-null

A nested equivalent is also acceptable:

type PendingSeatTaskItem = {
  task: Task;
  primary_item: Item | null;
  pending_upholstery_reason:
    | "missing_selection"
    | "missing_quantity";
  pending_upholstery: {
    item_upholstery_id: string | null;
    upholstery_id: string | null;
    amount_meters: number | null;
  };
  item_images: PendingTaskImage[];
};

The exact shape should follow existing backend serializer conventions.

Do not perform an item-upholstery request for every rendered card. That would create an N+1 request pattern and make direct actions dependent on secondary loading.

Acceptable solutions, in preferred order:

1. The pending-list response includes the active item_upholstery_id.
2. The pending-list response includes an existing item-upholstery summary object containing its client_id.
3. The amount update contract can validly identify the relation by item ID instead of relation ID, if such an existing backend contract already exists.
4. A single batched request provides the required relations for all visible rows.

A separate request per card is not acceptable.

If the actual backend response has already been extended with the identifier, consume the existing field and document its exact location in the implementation plan.

14. Optimistic update strategy

Optimistic behavior should be used only where the resulting transition is deterministic.

Backend calculations remain authoritative, especially because selecting upholstery may cause the task to transition from missing_selection to missing_quantity.

Use pending_upholstery_reason to determine the row’s pre-mutation bucket.

Mutation: create first item upholstery

This mutation resolves the missing_selection condition.

However, the item may immediately enter the missing_quantity bucket if the newly created relation has no amount.

Therefore:

* remove the row optimistically from missing_selection-only cached lists;
* decrement missing_selection_total, clamped at zero;
* do not blindly remove the row from every combined pending cache;
* do not assume the task has left all pending states;
* invalidate counts and pending lists after the mutation settles so the backend can report the cross-bucket transition.

For the first implementation, the safest behavior is:

optimistically update the exact active single-bucket list;
invalidate all pending list variants and counts after settlement.

Mutation: save positive upholstery amount

This mutation resolves missing_quantity.

It can optimistically:

* remove the row from missing_quantity-only lists;
* remove the row from combined pending lists;
* decrement missing_quantity_total, clamped at zero.

The task-detail and item-upholstery queries must still be invalidated because the backend may recalculate requirement state.

onMutate

For pending-page mutations:

* cancel active pending-list queries;
* cancel the counts query;
* snapshot affected list and count caches;
* mark only the affected row as pending;
* perform the safe optimistic list update;
* perform the safe optimistic count update.

onError

* restore all saved cache snapshots;
* restore the task row;
* restore counts;
* clear the row-level pending state;
* display mutation failure feedback.

onSuccess or onSettled

Invalidate canonical keys for:

* all pending-seat list variants;
* pending-seat counts;
* item upholstery for the affected item;
* task detail for the affected task;
* any existing task queries affected by upholstery requirement recalculation.

The implementation must not depend exclusively on optimistic cache writes. Backend-calculated requirement state must be reconciled through invalidation.
