# Intention: Reliable Cross-Platform Web Push Notifications
## Context
The frontend real-time layer has now been implemented across the applications.
The current real-time architecture uses Socket.IO as a live change-signal transport:
- workspace and user rooms are joined automatically by the backend;
- entity-specific presence is communicated through `view_entity` and `leave_entity`;
- socket payloads contain identifiers rather than complete entity data;
- TanStack Query remains the source of truth for frontend server state;
- `notification:new` is user-scoped and carries only the notification `client_id`;
- when the application is open, the frontend reacts to this event by refetching persisted notifications from the REST API;
- the backend already creates notification records independently of the socket connection.
The next phase is to implement the Web Push/VAPID notification system for cases where the application is closed, suspended, disconnected, or running in the background.
This must not replace or become coupled to the existing Socket.IO real-time layer. Socket.IO and Web Push are separate delivery channels operating from the same persisted notification domain.
Claude should inspect the existing backend notification architecture, task infrastructure, frontend notification package, PWA/service-worker setup, authentication lifecycle, shared-package boundaries, and relevant architectural contracts before designing the implementation plan.
---
## Primary intention
Build a reliable Web Push notification system that works across:
- Android browsers and installed PWAs;
- iPhone and iPad Home Screen web applications;
- supported desktop browsers;
- multiple devices and browser installations belonging to the same user.
The system should deliver operating-system notifications when the application is not actively open, while ensuring that the persisted notification record remains the canonical source of truth.
The expected high-level flow is:
```text
Business event
    ↓
Create persisted Notification record
    ↓
Commit the business transaction
    ↓
Dispatch independent delivery channels
    ├── Socket.IO signal for connected applications
    └── Durable Web Push delivery task
            ↓
       Load the user's active device subscriptions
            ↓
       Send encrypted VAPID push to each subscription endpoint
            ↓
       Browser/OS push infrastructure
            ↓
       Service worker is awakened
            ↓
       System notification is displayed

A failure in Web Push delivery must never prevent the notification from being stored or made available through the normal notification REST API.

⸻

Architectural direction

1. Persisted notification as the source of truth

The existing notification database record should remain authoritative.

Web Push should be treated only as a delivery mechanism. It must not introduce a second notification domain, duplicate business notification logic, or create push-only notification content that cannot later be retrieved from the application.

The push payload may contain enough information to display a notification while the app is closed, but it should reference the persisted notification and its related entity.

The system should preserve the existing distinction:

Application open:
notification:new socket signal
    → invalidate/refetch notification queries
    → obtain full content from REST
Application closed:
Web Push payload
    → service worker displays operating-system notification
    → notification click opens the application
    → application retrieves canonical state from REST

Both channels may be attempted for the same notification. They should not be considered mutually exclusive.

⸻

2. Multi-device subscription model

A user must be able to register multiple independent push subscriptions.

A subscription represents one browser installation or installed PWA, not the user as a whole. A single user may therefore have subscriptions for:

* an Android phone;
* an iPhone;
* a desktop browser;
* multiple browsers on the same device;
* multiple installed application surfaces.

Claude should design an appropriate backend subscription model that can safely store the complete browser-generated PushSubscription, including:

* endpoint;
* p256dh;
* auth;
* owning user;
* application or surface identity where relevant;
* creation and update timestamps;
* last confirmation or successful delivery time;
* failure state;
* disabled or revoked state;
* optional diagnostic metadata such as platform or user agent.

The endpoint should be treated as a unique subscription identity, and registration should be idempotent through an upsert rather than creating duplicates.

The design must also define what happens when:

* the same device registers again;
* a user signs out and another user signs in on the same installation;
* browser permission is revoked;
* a subscription expires or is replaced;
* a user disables notifications only for the current device;
* a user has multiple stale subscriptions.

⸻

3. Durable asynchronous delivery

Push delivery should not execute directly inside the business request or transaction that creates the notification.

After the notification is successfully persisted, a durable background task should be queued. The task should carry stable identifiers, such as the persisted notification identifier and recipient user identifier, rather than embedding stale subscription or notification snapshots.

The worker should reload:

* the notification;
* recipient delivery preferences;
* currently active subscriptions;
* current payload information;

before attempting delivery.

The implementation plan should determine how this fits into the backend’s existing task and command infrastructure.

Delivery jobs must be:

* retryable;
* observable;
* idempotent where possible;
* safe when executed more than once;
* isolated so one invalid device subscription does not block delivery to other devices.

⸻

4. Delivery lifecycle and failure classification

The system should distinguish between successful provider acceptance, transient delivery failures, permanent subscription failures, and configuration errors.

The implementation plan should define handling for at least:

* successful push-service acceptance;
* expired or invalid subscriptions, commonly represented by HTTP 404 or 410;
* rate limiting such as HTTP 429;
* provider/server failures such as HTTP 5xx;
* network timeouts;
* malformed subscription data;
* invalid VAPID configuration;
* payload encryption or serialization errors.

Permanent subscription failures should disable or remove the affected subscription without retrying it indefinitely.

Transient failures should use bounded exponential backoff and respect provider retry information where available.

Blind retries for invalid requests or broken application configuration should be avoided.

A delivery failure for one subscription must not prevent attempts to the user’s remaining subscriptions.

⸻

5. Delivery observability

The architecture should make it possible to trace the full push lifecycle.

Claude should determine whether delivery attempts require a dedicated delivery record or equivalent persistent audit model.

The system should be able to answer:

* Was the notification created?
* Was push delivery requested?
* Was the background task queued?
* Which subscriptions were targeted?
* Did the push service accept each request?
* Which attempts were retried?
* Which subscriptions were disabled as stale?
* What error or provider status was returned?
* When was the last successful delivery to a device?

Provider acceptance must not be described as proof that the user viewed the notification. The design should distinguish between:

* task queued;
* request attempted;
* push service accepted;
* permanent failure;
* notification clicked or application opened, if click tracking is later implemented.

Logs and records should avoid exposing private cryptographic subscription material.

⸻

Frontend subscription lifecycle

6. Explicit permission onboarding

Notification permission must only be requested after a clear user action.

The application should not request notification permission automatically during startup or immediately after sign-in.

The frontend should provide an understandable onboarding flow that explains why notifications are useful before showing the browser permission prompt.

Possible states should be represented explicitly:

Unsupported browser
Permission not requested
Permission denied
Permission granted but no subscription exists
Subscription exists locally but is not registered with the backend
Subscription registered and healthy
iPhone installation required
Backend registration failed
Notifications disabled for this device

The implementation should use feature detection for actual capability and platform detection only when necessary to provide platform-specific guidance.

⸻

7. iPhone and iPad reliability

The system must account for the requirements of Web Push on iOS and iPadOS.

The intended iPhone flow is:

User opens the website
    ↓
Application detects that it is running on iOS outside standalone mode
    ↓
User receives clear “Add to Home Screen” guidance
    ↓
User opens the installed Home Screen application
    ↓
User presses “Enable notifications”
    ↓
Application requests notification permission
    ↓
Application creates and registers the PushSubscription
    ↓
Application confirms the complete delivery path with a test notification

The implementation plan should verify current Safari and iOS requirements during research rather than relying only on assumptions in this intention.

The frontend must not misleadingly present notifications as enabled merely because permission is granted. It should confirm that:

* the app is in a supported environment;
* a service worker is active;
* a PushSubscription exists;
* the subscription was successfully registered with the backend.

The design should include clear recovery paths when the user:

* has not installed the PWA;
* denied permission;
* deleted and reinstalled the Home Screen application;
* cleared Safari website data;
* changed notification settings in iOS;
* granted permission but no valid subscription remains.

⸻

8. Android reliability

Android should use the same Web Push domain and backend APIs, while allowing its simpler supported-browser and PWA installation flows.

The implementation should still avoid relying on browser-specific assumptions.

It should support:

* installed PWAs;
* supported browser tabs where Web Push is available;
* application restart;
* browser update;
* subscription refresh;
* user sign-out and sign-in;
* multiple Android devices.

Android-specific testing should include cases where the application is:

* foregrounded;
* backgrounded;
* fully closed;
* removed from recent applications;
* reopened after missing several socket events.

The application must recover canonical notification state through REST when reopened, regardless of whether an operating-system notification was successfully displayed.

⸻

9. Subscription reconciliation

The frontend should not rely only on the initial Enable Notifications action or on browser lifecycle events.

Whenever appropriate, such as after authentication, application startup, service-worker readiness, or service-worker activation, the frontend should reconcile the browser’s current subscription with the backend.

The high-level reconciliation logic should be:

Check platform support
    ↓
Inspect notification permission
    ↓
Wait for active service-worker registration
    ↓
Read pushManager.getSubscription()
    ↓
Compare or upsert against backend registration
    ↓
Expose an accurate device notification status

The implementation plan should define when automatic reconciliation is safe and when a user action is required.

It should also evaluate support for pushsubscriptionchange, while treating startup reconciliation as the primary recovery mechanism because lifecycle-event support may vary across browsers.

⸻

10. Sign-out and user switching

Push subscriptions must not accidentally remain associated with the wrong user.

The implementation plan should define a reliable sign-out policy.

At minimum, it should consider:

* unregistering or disassociating the current device subscription from the signed-out account;
* preserving the browser-level subscription if it can safely be reassigned on the next sign-in;
* preventing notifications intended for the previous user from being delivered after account switching;
* handling sign-out while offline;
* retrying backend cleanup or reassociation when connectivity returns.

The approach should align with the current authentication and shared-package architecture.

⸻

Service-worker behavior

11. Minimal push-event handler

The service worker should be able to display a useful notification without depending on:

* React;
* the application UI;
* an active authentication token;
* an HTTP request;
* Socket.IO;
* TanStack Query;
* complex IndexedDB initialization.

The push handler should:

1. parse a versioned payload;
2. fall back safely when data is missing or malformed;
3. call showNotification;
4. wrap asynchronous work in event.waitUntil;
5. finish quickly.

A malformed payload should not cause the push event to fail silently.

The service worker should not treat push delivery as a place to reproduce the frontend domain layer.

⸻

12. Versioned and resilient payload contract

The push payload should have a stable, explicit schema and version.

A possible conceptual payload is:

{
  "version": 1,
  "notification_client_id": "...",
  "title": "...",
  "body": "...",
  "url": "/notifications/...",
  "entity_type": "...",
  "entity_client_id": "...",
  "tag": "notification:..."
}

Claude should research the practical encrypted Web Push payload-size limits and design the smallest sufficient contract.

The payload must not contain:

* access tokens;
* confidential notes;
* unnecessary personal information;
* complete entity records;
* internal-only identifiers when public client identifiers are available.

The system must account for lock-screen exposure even though Web Push transport payloads are encrypted.

Older service workers should remain able to process newer backend payloads safely through version checks and fallback behavior.

⸻

13. Notification deduplication and replacement

The service worker should use stable notification tags to control duplicate operating-system notifications.

The implementation plan should distinguish between:

* independent notifications that should coexist;
* repeated delivery attempts for the same persisted notification;
* state notifications where a newer event should replace an older one.

A notification-level tag can make retries visually idempotent:

notification:{notification_client_id}

Entity or category tags may be used only where replacement semantics are intentional.

The system must avoid one global tag that causes unrelated notifications to overwrite each other.

⸻

14. Notification-click navigation

Clicking a system notification should reliably open or focus the correct application surface.

The service worker should:

* close the system notification;
* resolve a same-origin durable URL;
* search existing controlled and uncontrolled application windows;
* focus an appropriate existing window when possible;
* navigate it to the desired route when safe;
* otherwise open a new application window;
* wrap the complete operation in event.waitUntil.

Navigation should use durable application routes, preferably through the persisted notification identifier, rather than encoding fragile frontend state in the push payload.

A suitable conceptual flow is:

User clicks notification
    ↓
Open or focus /notifications/{notification_client_id}
    ↓
Application loads persisted notification from REST
    ↓
Application resolves its current canonical entity route
    ↓
Navigate to task, case, conversation, or other target

This should remain valid if route structures evolve or if the referenced entity no longer exists.

⸻

Delivery policy and user preferences

15. Notification channel policy

The notification domain should decide which delivery channels apply. The Web Push sender should execute the decision rather than contain business-category rules.

The implementation should support future or current preferences such as:

task assigned:
    in-app = true
    push = true
inventory informational update:
    in-app = true
    push = false
critical failure:
    in-app = true
    push = true

Claude should inspect whether notification preferences already exist and design the appropriate extension without unnecessarily expanding the first implementation.

Push suppression based on current entity presence should be considered carefully.

A socket connection alone must not be treated as evidence that the user saw a notification. The user may be connected on another device, on another screen, in the background, or with the device locked.

For important notification categories, the architecture should permit push delivery even when a socket connection exists.

⸻

16. Device health and test notification

The frontend should expose the real push status for the current device.

A user-facing status surface should be able to communicate:

Notifications supported
Permission allowed or denied
Current device subscribed
Backend registration confirmed
Last registration confirmation
Notifications disabled on this device
Installation required on iPhone

The implementation should include a test-notification action that validates the complete chain:

Frontend request
    ↓
Backend creates or dispatches a test push
    ↓
Background delivery task
    ↓
Push provider
    ↓
Service worker
    ↓
Operating-system notification

A successful subscription API response alone is not sufficient to prove that push delivery works.

The test path should reuse production delivery infrastructure rather than introducing a separate direct-sending implementation.

⸻

Shared package and application boundaries

Claude should investigate the correct ownership of each concern.

A likely separation is:

Backend notification domain
├── persisted notifications
├── notification policy
├── push subscription model
├── subscription registration APIs
├── delivery task creation
├── Web Push sender
├── delivery-attempt tracking
└── stale-subscription cleanup
Shared frontend notifications package
├── capability detection
├── permission and subscription lifecycle
├── backend registration/reconciliation
├── device notification status
├── test notification action
└── reusable notification settings primitives
PWA/service-worker package or application worker
├── push event handling
├── notification display
├── payload-version handling
└── notification-click routing
Applications
├── user-facing onboarding placement
├── iPhone installation guidance
├── settings-screen integration
├── app-specific deep-link resolution
└── sign-out lifecycle integration

This is a direction rather than a fixed file structure. Claude should align the final implementation plan with the repository’s current shared-package contracts and avoid duplicating push lifecycle logic independently across managers, workers, and future seller applications.

The VAPID private key must remain backend-only.

The VAPID public key may be exposed through the existing public endpoint or configuration mechanism, but the architecture should define key loading, validation, rotation implications, and environment-specific setup.

⸻

Reliability and recovery principles

The implementation should be guided by these principles:

1. Persist the notification before attempting any delivery channel.
2. Push delivery must be asynchronous and retryable.
3. Socket.IO and Web Push are independent transports.
4. A user may have multiple subscriptions.
5. Each subscription can fail independently.
6. Permanent invalid subscriptions should be retired automatically.
7. Permission granted does not necessarily mean the backend has a valid subscription.
8. Startup reconciliation should repair drift.
9. Service-worker handlers must be minimal and self-contained.
10. Push payloads must be versioned, small, and non-sensitive.
11. REST must restore canonical state after missed or delayed pushes.
12. Provider acceptance does not prove user visibility.
13. A failed push must never lose the persisted notification.
14. The same infrastructure should support Android, iPhone, desktop, and future applications.
15. Reliability must be demonstrated through end-to-end testing, not assumed from successful registration.

⸻

Validation expectations

Claude should produce an implementation plan with automated and manual validation for the complete system.

Validation should include at least:

Backend

* subscription creation and idempotent upsert;
* multiple subscriptions for one user;
* reassignment or cleanup during user switching;
* VAPID configuration validation;
* payload serialization and encryption;
* successful provider acceptance;
* one stale subscription alongside one healthy subscription;
* permanent 404/410 cleanup;
* transient 429, timeout, and 5xx retries;
* retry exhaustion;
* duplicate task execution;
* notification persistence despite delivery failure;
* delivery-attempt observability;
* test-notification path.

Frontend

* unsupported browser state;
* permission not requested;
* permission denied;
* permission granted;
* granted permission with no subscription;
* successful subscription creation;
* backend registration failure;
* startup reconciliation;
* stale local/backend subscription mismatch;
* sign-out cleanup;
* sign-in as a different user;
* service-worker update;
* disable notifications for the current device;
* test-notification status.

Service worker

* valid payload display;
* malformed payload fallback;
* unknown payload version;
* duplicate delivery using the same tag;
* notification click with an existing app window;
* notification click with the app closed;
* deleted or unavailable target entity;
* same-origin navigation enforcement.

Physical-device validation

Real-device testing is required because browser emulation alone cannot validate operating-system push delivery.

At minimum:

* current supported Android device/browser;
* Android installed PWA;
* iPhone running a currently supported iOS version;
* iPhone Home Screen-installed PWA;
* application foregrounded;
* application backgrounded;
* application fully closed;
* device locked;
* network temporarily unavailable;
* application reopened after missed notifications;
* multiple devices registered for the same user;
* permission revoked through operating-system settings.

Claude should verify current platform requirements, Web Push limitations, service-worker behavior, and practical payload constraints using authoritative browser and platform documentation while developing the detailed plan.

⸻

Expected outcome

The final system should provide a dependable notification experience where:

* users receive system notifications on supported Android and iPhone installations when the application is closed;
* open applications continue using the existing Socket.IO and REST-based notification flow;
* every notification remains available through the persisted notification list even when push delivery fails;
* devices can register, reconcile, disable, expire, and recover independently;
* stale endpoints are automatically retired;
* transient failures are retried safely;
* clicking a notification opens the correct application context;
* notification status is understandable and testable by the user;
* delivery behavior is observable by developers;
* managers, workers, and future applications reuse the same shared infrastructure without duplicating platform-specific logic.
